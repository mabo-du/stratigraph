# From Mud to Metadata: A Practical Framework for Resilient, Cryptographically-Auditable Collaborative Archaeology in the Field

## Ensuring Graph Integrity with CRDT-Compatible Transitive Reduction
The primary challenge in integrating the Dye & Buck transitive reduction algorithm into StratiGraph's Yjs-based architecture is reconciling its requirement for a global graph view with the inherently concurrent and decentralized nature of Conflict-Free Replicated Data Types (CRDTs). For an archaeological Harris Matrix—a legally and scientifically defensible record of a site destroyed by excavation—the correctness of its topological structure is paramount; any divergence or introduction of cycles during a merge event represents a catastrophic failure of data integrity. Therefore, the research must prioritize practical implementation patterns that provide strong safety guarantees, anchored by just enough theoretical understanding to ensure convergence and preserve the acyclic, minimal properties of the DAG. The analysis indicates that direct integration of the reduction algorithm as a single CRDT operation is infeasible due to its non-local dependency on the complete graph state [5]. Instead, the most robust strategy involves decoupling the reduction logic from the real-time edit operations and executing it as a deterministic, post-merge reconciliation phase. This approach trades immediate visual feedback for guaranteed structural consistency across all peers.

The fundamental obstacle to making transitive reduction a CRDT operation lies in its definition. Transitive reduction aims to find the smallest subset of edges in a DAG that preserves the same reachability relationships as the original graph [5]. To determine if an edge (u, v) is redundant, one must check if there exists an alternative path from u to v of length two or more. This query requires knowledge of the entire graph topology, making the operation inherently non-local and non-commutative. A CRDT operation, by contrast, must be commutative, associative, and idempotent to ensure that the order of application does not affect the final converged state. Since applying transitive reduction to a partial, inconsistent view of the graph would yield different results depending on the merge order, it cannot be expressed as a primitive CRDT operation. Furthermore, while advanced concepts like reduction as a monotone lattice operation have been explored in formal methods, they face significant challenges in this context. While removing edges can be considered a "less connected" state, a subsequent merge could reintroduce an edge that was previously deemed redundant, complicating strict monotonicity and making such an approach less reliable than a simpler, deterministic model [2, 7].

Given these constraints, the most practical and safe implementation strategy is deferred post-merge reduction. This pattern treats transitive reduction not as a continuous background process but as a discrete, corrective action performed only after a full merge event has completed. The workflow involves allowing peers to make concurrent edits using standard Yjs operations (Y.Map, Y.Array) without interference. When a merge is detected, the system enters a "reduction phase." First, it performs a full topological sort on the merged graph to verify that no cycles were introduced. If a cycle is found, it is quarantined or flagged for user intervention, adhering to best practices for managing DAGs within CRDTs [3]. Assuming the graph remains acyclic, the full Dye & Buck algorithm is then executed deterministically on the consolidated, converged graph. The resulting minimal set of edges is then submitted back to the Yjs document as a single, atomic batch of operations. This ensures that every peer applies the exact same reduction, guaranteeing perfect convergence. While this introduces latency between a user's edit and the visible update to the minimal graph, it provides an unshakeable guarantee of correctness. The "just enough theory" to support this is the understanding that transitive reduction is a total function over the graph's state space; when applied to the same input (the fully merged, acyclic graph), it will always produce the same output.

This deferred approach aligns well with the existing HMDP schema and archaeological workflows. The temporary presence of redundant edges after an edit is made locally is unlikely to cause confusion, as the primary user interface focuses on the logical relationships rather than the raw graph representation. The critical moment for integrity verification occurs at the merge point, which is precisely where the deferred reduction is triggered. An alternative, more complex strategy would involve leader-elected reconciliation, where one peer is designated to perform the reduction. However, in a zero-infrastructure, true P2P environment, electing a leader adds significant complexity and potential points of failure, making it a poor fit for this use case. There are no known native CRDT graph types beyond foundational structures like OR-Sets that offer built-in preservation of DAG minimality, reinforcing the need to implement this logic as a separate, post-processing step [1]. The implementation would require careful memory management, as Yjs retains operation history indefinitely, which could become problematic for very large documents [17].

| Strategy | Convergence Guarantee | Complexity | User Experience Impact | Suitability for StratiGraph |
| :--- | :--- | :--- | :--- | :--- |
| Direct Algorithm Integration | Low; Non-commutative and non-local. | High; Requires novel CRDT construction. | None; Not feasible. | Unsuitable |
| Reduction as Monotone Lattice | Moderate; Complicated by merges reintroducing edges. | High; Theoretically complex. | Minimal; Potential for subtle bugs. | Unproven and risky. |
| Leader-Elected Reconciliation | High; Depends on leader stability. | Very High; Adds coordination overhead. | Poor; Introduces a single point of failure. | Unsuitable for P2P. |
| Deferred Post-Merge Reduction | High; Deterministic on a converged state. | Moderate; Requires merge detection logic. | Delayed; Visual feedback is not instantaneous. | Highly Suitable |

In conclusion, the recommended path for StratiGraph v3 is to implement deferred post-merge transitive reduction. This strategy provides the strongest possible guarantee of data integrity by performing a deterministic, globally-aware cleanup only after all concurrent changes have been successfully merged. It balances the high-stakes requirement for a correct Harris Matrix with the practical realities of a distributed, offline-first application, forming a cornerstone of the system's reliability.

## Robust Offline-First Binary Synchronization for Field Media
Syncing large binary attachments, such as high-resolution photographs of stratigraphic contexts, presents a formidable challenge in zero-infrastructure field environments characterized by unstable and intermittent network connectivity. Standard transfer protocols often fail under these conditions, leading to incomplete files and frustrating user experiences. The solution requires a multi-faceted approach combining content-addressable storage (CAS) for deduplication, a custom resumable and idempotent transfer protocol over WebRTC data channels, and a precise metadata model within the Yjs document to inform the user interface about attachment availability. This integrated system prioritizes robustness and data completeness above all else, ensuring that media assets are transferred reliably even when connections drop repeatedly—a common occurrence on a busy dig site.

The foundation of a robust binary sync system is content-addressable storage. Each binary blob (photo, artefact file) is hashed using a cryptographic function like SHA-256, and the resulting hash becomes its unique address [19]. Before initiating a transfer, a peer first checks its local storage for a file with that hash. If the file exists, no transfer is needed. If it does not, the peer requests the file from other peers who advertise having it. This CAS model provides several critical benefits for the archaeological context. Firstly, it eliminates redundant transfers of identical files across the team, conserving precious bandwidth and time. Secondly, it provides a strong guarantee of data integrity; upon receiving a chunk, the recipient can recompute its hash to verify it matches the expected value. Systems like IPFS and Blossom demonstrate the viability of this paradigm in distributed systems, and a custom implementation within StratiGraph's Tauri app would provide the necessary control and integration with the SQLite/IndexedDB hybrid storage backend [1].

To handle the physical unreliability of WebRTC connections, the transfer protocol itself must be designed for resilience. A simple chunked transfer mechanism is effective, where each blob is broken into smaller, fixed-size chunks. The protocol must be both resumable and idempotent. Resumability is achieved by including metadata with each chunk request: the blob's root hash, the total size of the blob, and the specific chunk index or byte offset being requested. Idempotency is ensured by designing the protocol so that requesting the same chunk multiple times results in the same outcome. The sender simply re-transmits the requested chunk whenever asked. The receiver maintains a manifest or a bitfield tracking which chunks of a given blob it has successfully received. When a connection drops, the transfer can be resumed from the last known good chunk by querying the sender for the next missing piece. This design directly addresses the core field problem where a tablet might go to sleep or someone walking between devices causes a dropped connection mid-transfer. For payloads in the 5-50 MB range, complex delta-sync algorithms are likely overkill compared to the simplicity and reliability of this straightforward chunked approach.

A crucial aspect of this system is providing fine-grained awareness of attachment availability to the user. An archaeologist must be able to distinguish whether a photo for Context SU047 is present on the device, or merely available on a colleague's device, before deciding whether to reference it in their notebook. This requires a clear metadata model stored within the Yjs document alongside the context object. Two pieces of information are essential:

- `attachmentHash`: A string property containing the cryptographic hash of the media file. This property is populated when another peer shares a context with an attachment.
- `isAttachmentAvailableLocally`: A boolean flag that reflects the current state of the local file system. This flag starts as false.

When Peer A sends a context object to Peer B, Peer B's Yjs document receives the context object with the `attachmentHash` populated. At this point, the UI can display a placeholder icon indicating that an attachment exists but is not yet available. Simultaneously, a background worker in Peer B's Tauri app, listening to the Yjs updates, detects the new hash and initiates a download request via the binary sync protocol. Once the file is completely downloaded and verified against the hash, the worker updates the `isAttachmentAvailableLocally` flag to true. The UI then triggers a re-render, displaying the actual image. This two-state model cleanly separates the logical existence of an attachment from its physical presence on the local device, empowering the user with accurate information for their work. For the transfer medium, WebRTC Data Channels are appropriate. Using the unreliable datagram channel with a custom protocol built on top allows for efficient, low-latency transfers, though care must be taken to implement the necessary resumption logic at the application layer.

| Custom Chunked Protocol | Files are split into chunks. Transfers are resumable and idempotent. | Robustness against dropped WebRTC connections; ensures complete file transfers. [10] |

By combining these elements—a CAS foundation, a resilient transfer protocol, and a detailed metadata model—StratiGraph v3 can achieve a level of robustness in binary media synchronization that is fit for the demanding conditions of archaeological fieldwork.

## A Zero-Trust Identity Architecture for Secure Peer-to-Peer Collaboration
In a zero-infrastructure setting, the absence of a central server or certificate authority necessitates a fundamentally different approach to trust. For StratiGraph, where the data being shared is not just digital information but a legally binding record of cultural heritage, security cannot be an afterthought. The threat model includes opportunistic adversaries like nighthawkers who may attempt to passively eavesdrop on network traffic or actively inject malicious data, as well as the risk of device theft, which would lead to a complete compromise of all project data. To counter these threats, StratiGraph v3 must adopt a zero-trust architecture where every peer is cryptographically identified, every data exchange is authenticated, and all data is protected both in transit and at rest. The optimal mechanism for establishing this trust in the initial "first connection" scenario is a QR-code-rooted key ceremony, which simultaneously solves the problems of authenticated peer admission, immutable auditability of mutations, and resistance to physical-world adversaries.

The cornerstone of this architecture is a mandatory QR-code-based key ceremony for any new peer wishing to join a project. This process transforms a purely technical cryptographic procedure into a tangible, physically-verifiable action suitable for non-technical archaeologists. Upon first launch, each device generates a self-sovereign Ed25519 keypair, creating a unique, persistent identity [13]. When two archaeologists want to collaborate, one device displays its public key as a scannable QR code. The second archaeologist uses their device to scan the code. This act of scanning, combined with verbal confirmation between the users ("Is that your name?"), establishes a shared secret or confirms the identity of the other party. This ceremony serves as the cryptographic root of trust. Based on this, a secure session key can be derived using a Password-Authenticated Key Exchange (PAKE) protocol like SPAKE2, which protects against man-in-the-middle attacks even if the initial QR code is spoofed [15]. This session key is then used to encrypt all subsequent mDNS discovery broadcasts and other network traffic, preventing passive observers from inferring site activity or topology.

This established identity is the key to achieving two other critical goals: auditability and authenticated peer admission. Every mutation to the Yjs document—every creation, update, or deletion of a context or relationship—must be cryptographically signed by the private key corresponding to the device's identity. The application layer must intercept outgoing Yjs update blobs, encode them, and generate a detached signature using the device's Ed25519 private key. This signature is transmitted alongside the update. On the receiving end, before applying any update to the local document, the application verifies the sender's signature using their stored public key. This process creates an immutable, append-only log of all actions, proving the provenance of every change and satisfying chain-of-custody requirements for publication and legal compliance [7, 20]. This same signature verification process enforces authenticated peer admission: any update from a device whose signature cannot be verified is rejected outright, acting as a powerful defense against rogue peer injection. No meaningful data exchange, whether it be Yjs document updates or binary media, should ever occur until a successful key ceremony has been completed and a verified identity is established.

Finally, this architecture provides strong resistance to opportunistic adversaries. The QR-code ceremony requires physical co-presence and line-of-sight, meaning an attacker cannot complete the process without being visibly present among the legitimate team members. This deters casual intrusion. More importantly, all local storage paths—including SQLite for the Yjs document and IndexedDB for binary blobs—must be protected with strong encryption at rest, such as AES-256 [10]. This measure ensures that even if a device is stolen, the data remains inaccessible, protecting sensitive information about the excavation site's layout and findings. The combination of encrypted discovery, signed operations, and encrypted-at-rest storage creates a multi-layered defense that is both robust and practical for the field. The QR-code ceremony is the elegant linchpin that ties all these security measures together, providing a usable onboarding flow that delivers significant security benefits without overwhelming the user.

| Security Goal | Mechanism | Implementation Details | Threat Mitigated |
| :--- | :--- | :--- | :--- |
| Authenticated Peer Admission | Mandatory QR-Code Ceremony & Signature Verification | Devices generate Ed25519 keypairs. A ceremony establishes a shared secret. All data exchanges are refused without a valid signature. | Rogue peer injection into the collaboration network. |
| Mutation Auditability | Cryptographic Signing of Yjs Operations | Each Yjs update is signed with the device's private key before transmission. Recipients verify signatures before applying updates. | Deniability of actions; loss of chain-of-custody for legal and scientific records. [20] |
| Resistance to Passive Eavesdropping | Encrypted Discovery Protocols | mDNS or Bluetooth advertisements are encrypted using a session key derived during the QR-code ceremony. | Inferring site activity or network topology by sniffing unencrypted traffic. [10] |
| Data-at-Rest Protection | Full-Disk/File Encryption | All local storage (SQLite, IndexedDB) is protected with AES-256 encryption. | Complete data loss or misuse in the event of device theft. |

This zero-trust architecture, rooted in a simple yet powerful key ceremony, provides the necessary confidence for archaeologists to collaborate securely in the field, knowing that the integrity and confidentiality of their data are rigorously protected.

## Integrated System Design and Cross-Cutting Principles
The successful implementation of StratiGraph v3 hinges on the seamless integration of its core architectural components—CRDT-compatible transitive reduction, robust binary synchronization, and a zero-trust identity system—within a cohesive whole. These elements are not isolated features but deeply interconnected subsystems that rely on shared principles to deliver a product that is trustworthy, reliable, and usable in the extreme conditions of archaeological fieldwork. The overarching design philosophy is one of layered responsibility and explicit resilience, where each component is designed to handle the failures inherent in a zero-infrastructure environment while respecting the absolute priority of data integrity. This synthesis reveals several cross-cutting themes that define the application's architecture and ensure its suitability for producing legally defensible scientific records.

A primary principle governing the architecture is the layering of concerns. The system is deliberately modular, separating the logical state of the Harris Matrix, the physical storage of media, and the cryptographic security layer. The Yjs document acts as the authoritative source of truth for the graph's structure (nodes and edges) and the associated metadata for contexts, including the `attachmentHash` and availability flags. This logical layer is agnostic to how large blobs are managed. A parallel, independent service handles all binary synchronization, communicating with the Yjs layer via events and notifications. Finally, a higher-level security framework sits atop the communication layer, responsible for key ceremonies, message signing, and encryption. This separation is crucial for maintainability, testability, and scalability. For instance, the performance characteristics of Yjs for representing graph nodes can be optimized independently of the blob sync protocol, and the cryptographic identity scheme can be updated without affecting the core graph logic. This modularity ensures that a failure or optimization in one area does not cascade uncontrollably into others.

The second, and perhaps most critical, principle is treating data integrity as a first-class citizen. Across all three major components, the design choices are driven by the imperative to prevent data corruption or loss. The deferred post-merge transitive reduction ensures the mathematical correctness of the Harris Matrix DAG. The CAS-based, resumable binary transfer protocol guarantees the completeness and authenticity of attached media files. The cryptographic signing of every Yjs operation and the encryption of all data exchanges provide an auditable trail of provenance and protect against tampering and unauthorized access. This holistic approach means that integrity is not an ad-hoc feature but is woven into the fabric of the system's operations. It acknowledges that in the context of heritage preservation, a corrupted or inconsistent dataset is not merely a bug but an irreversible loss of irreplaceable information. The system is architected to make the correct, consistent state the default and the only achievable state.

Third, the architecture embraces the reality of the operational environment by designing for imperfection with guardrails. The field is not a controlled lab; it is a dynamic, unpredictable space with dropping Wi-Fi signals, sleeping tablets, and muddy hands. Rather than fighting these realities, the design accommodates them gracefully. The resumable binary transfer protocol explicitly accounts for dropped WebRTC connections. Background workers for media syncing prevent slow transfers from freezing the user interface. The deferred reduction process accepts that merges take time and prioritizes correctness over immediacy. Safety is not achieved by assuming perfect network conditions, but by building explicit mechanisms to handle their absence. Guardrails, such as the mandatory QR-code ceremony, are implemented to prevent users from accidentally compromising the system's security. This pragmatic approach ensures that the application remains functional and reliable even when its operating environment is hostile.

Finally, the design demonstrates a commitment to security as usability. The choice of a QR-code-based key ceremony is a prime example of this principle in action. It translates a complex cryptographic concept into a simple, tangible user action—"scan this code"—that is intuitive for a non-technical domain expert standing in a trench. This single mechanism elegantly solves multiple, seemingly unrelated problems: it provides a secure way to bootstrap a trusted relationship, it enables forensic-level auditability, and it creates a meaningful barrier against opportunistic adversaries. By aligning security requirements with the user's mental model and physical actions, the system avoids the pitfall of introducing overly complex security measures that users would either ignore or misuse. The result is a system that is not only secure but also easy to operate correctly.

In synthesis, StratiGraph v3 is not merely an incremental update but a fundamental redesign grounded in the specific, high-stakes requirements of its domain. By adopting a layered architecture, prioritizing data integrity, embracing environmental unreliability, and aligning security with usability, the proposed strategies form a robust and trustworthy system. This architecture is capable of producing the kind of legally defensible, scientifically sound records that are essential for modern archaeology, even in the most challenging zero-infrastructure field environments.
