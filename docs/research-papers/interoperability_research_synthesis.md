# Synthesis: Local-First Interoperability in the Digital Heritage Ecosystem

Based on the two Deep Research reports, there is a strong consensus on the optimal architecture for seamlessly passing data between independent local tools (like Paleo Chrono, StratiGraph, Cache & Carry, and Trowel) without sacrificing your offline-first, local-first principles. 

Here is the synthesized architectural blueprint for the ecosystem.

## 1. The Core Data Bus: Shared Filesystem (`.heritage`)
Both reports strongly recommend against using local HTTP servers (due to CSRF/CORS vulnerabilities and port conflicts) or a monolithic SQLite database (due to WASM/OPFS origin isolation and severe file-locking contention).

**The Solution:**
- Use a dedicated **`.heritage` hidden directory** in the user's home folder as a universal, asynchronous data bus.
- **PWAs** (like StratiGraph) access this folder via the **File System Access API** (saving the `FileSystemDirectoryHandle` in IndexedDB for persistence).
- **Native Tauri Apps** (like Paleo Chrono) access it via standard OS filesystem APIs.
- Reactivity is achieved using the **FileSystemObserver API** (or native file watchers), allowing an open app like Trowel to instantly update its UI when a new file appears in the `.heritage` folder.

## 2. App Invocation & Handoff: URI Deep Linking
To provide a frictionless "App Suite" UX (e.g., a "Send to Trowel" button in Paleo Chrono), the ecosystem should use **URI Schemes / Deep Linking** combined with the shared filesystem.
- **Workflow:** Paleo Chrono saves a file (e.g., `.heritage/chronology/model.jsonld`), then invokes `trowel://import?ref=chronology/model.jsonld`. The OS routes this URI to Trowel, which reads the file from the shared workspace and injects it into the active report.

## 3. Data Format & Ontology: Flattened JSON-LD
To ensure interoperability without suffocating the apps in complex RDF graphs, the reports recommend a **Flattened JSON-LD Profile**.
- All data artifacts are stored as flat, atomic `.jsonld` files.
- The ecosystem uses a custom, lightweight namespace (e.g., `heritage:`) for simple developer ergonomics (`"stratigraphic_unit": "Context 45"`).
- The JSON-LD `@context` silently maps these flat keys to rigorous international standards:
  - **CIDOC CRMarchaeo** for stratigraphy and excavation events.
  - **Darwin Core (DwC)** for biological/fossil specimens (e.g., Paleo ID outputs).
  - **W3C PROV-O** for data provenance.

## 4. Provenance & Conflict Resolution
When an AI (Paleo ID) and a human curator (in Cache & Carry) interact with the same specimen, provenance and conflict mitigation are critical.
- **W3C PROV-O:** Embed provenance directly in the JSON-LD. Use `prov:SoftwareAgent` to denote an AI identification (along with confidence scores) and `prov:Person` for human curation.
- **Append-Only Assertion Logs:** Do not use simple versioned records where the last writer wins. Instead, the core physical entity is immutable, and scientifically volatile attributes (like taxonomic IDs) are an array of **Assertions**. When a human corrects an AI, they append a new assertion rather than deleting the AI's structural data.
- **Pessimistic File Conflicts:** If two apps write to the same file simultaneously, create a `.sync-conflict` file (inspired by Syncthing) rather than overwriting, allowing the user to manually merge differences later.

## 5. Schema Versioning for Solo Developers
With 6–8 independent apps, schema drift is inevitable. The recommended strategy is:
- **SchemaVer (Model-Revision-Addition):** Embed a semantic schema version (e.g., `"@version": "1-2-0"`) at the root of every JSON-LD payload.
- **Unknown Key Preservation:** This is the most vital technical safeguard. When an older app parses a file generated by a newer app, it *must not strip unrecognized keys*. It must hold them in a memory buffer and perfectly re-serialize them when saving. This guarantees forward compatibility and prevents silent data destruction.
