# Harris Matrix Tools: User Pain Points, Desired Features, and Future Directions

## Core Frustrations with Legacy Software  
Archaeologists report that existing Harris Matrix tools are often **outdated or brittle**.  Many legacy programs (e.g. ArchEd, Stratify, Bonn’s BASP) were designed decades ago and have received little maintenance.  For example, Stratify “has not been updated in 10 years” and still uses proprietary, closed code【27†L1470-L1478】; ArchEd’s last release was 2004【23†L278-L283】.  Users find these old tools hard to install or run on modern systems (Windows 10/11, macOS) and prone to crashes on large datasets【16†L76-L80】【27†L1470-L1478】.  

**Poor usability and scalability** are common complaints. Data entry often feels cumbersome: early reviews noted ArchEd requires tedious dialog entry of relationships【16†L45-L54】【23†L250-L259】.  Complex matrices become visually cluttered, and users lack control of layout. One practitioner noted “once the matrix gets very complicated, there is no way of taking control of how the matrix is presented… it ended up… rather un-aesthetic, with interconnecting lines all over the place”【27†L1470-L1478】.  In practice this leads many to “settle on something else” or to redraw the matrix in generic graphics software【27†L1470-L1478】.  

Because of these limitations, **archaeologists often bypass specialized software** entirely.  Many analysts build the matrix “by hand” or using tools like Excel, PowerPoint or CAD.  In practice, spreadsheets or drawing tools are common workarounds: one survey found practitioners using Excel for “working” matrices and only importing a final version into ArchEd/Stratify for visualization, then redrawing in Illustrator or diagrams.net for publication【27†L1491-L1500】【41†L127-L130】.  As one user commented: “you can make a Harris matrix in any software that lets you make diagrams… I have used PowerPoint and yED graph editor… This can make the data difficult to read and reuse, though”【41†L127-L130】.  In summary, archaeologists gravitate to generic graphing or CAD programs because legacy HM tools are seen as **inflexible, poorly supported, and expensive**, failing to meet the needs of modern complex projects【27†L1470-L1478】【41†L127-L130】.

## Feature Gaps and User Workarounds  
Practitioners explicitly request a range of features missing from legacy tools.  Chief among these is **integration with other archaeological datasets**.  Users want their Harris Matrix tied to finds databases, GIS spatial data, and stratigraphic databases.  For example, large firms store contexts in relational databases (e.g. MOLA’s Oracle system) and expect matrix software to link directly to those records.  In reality, strat relations are often extracted manually (e.g. exported to Bonn or ArchEd and then post-processed) because existing tools lack built-in database connectivity【37†L1010-L1017】.  Similarly, archaeologists want context boxes in the matrix to connect with geospatial features – e.g. clicking a context to highlight its layers on a site plan or 3D model – but most HM tools have no GIS interface【23†L270-L278】【31†L287-L295】.  

**Chronological and phasing support** is another gap.  Users ask for automatic export of matrix sequences into Bayesian-dating formats (OxCal/BCal, etc.) and for built-in time axes.  Modern practice increasingly combines stratigraphy with radiocarbon modeling, but “suitable form for input into Bayesian… software (OxCal, Chronomodel)” is still a manual process【35†L308-L317】.  Likewise, users struggle with inconsistent definitions of “phases” and “periods” in software: e.g. older tools may force a linear layering of units into phases, conflicting with how archaeologists conceptually group contemporaneous contexts【23†L297-L304】.  

**Large-scale urban sites** strain current systems.  One MOLA archaeologist described “deeply-stratified urban sites” where the “rigid and unwieldy nature of… stratigraphic matrices” is a “big challenge”【37†L1010-L1017】.  Matrices with thousands of units (e.g. 5000+ contexts) require features like collapsing/expanding groups, subgraphs, or phased sequences – but few tools handle this elegantly.  Scalability problems were noted even in early reviews (ArchEd struggled with ~1000 contexts)【16†L76-L80】.  

Given these deficits, users hack solutions or use hybrid workflows.  For instance, open-source initiatives like **Strati5** employ a spreadsheet-based mobile app: it records contexts in Excel on phones, checks for logical loops, and even supports real-time multi-user editing【65†L113-L121】.  This reflects demand for **mobile and collaborative** tools that legacy Windows-only apps lack.  Other groups script data conversions (e.g. Python scripts to transform excavator outputs into `.LST` or graph formats) to feed into Harris matrix builders.  

## Interoperability and Emerging Standards  
Modern archaeologists expect stratigraphic data to interoperate with broader tools and standards.  **GIS and spatial tools** are ubiquitous: teams often want to overlay stratigraphic units on site maps or 3D models in QGIS, ArcGIS or similar.  However, most HM programs remain siloed.  As one survey observed, “the Harris Matrix is usually part of [the] information amalgam, but outside the archaeological database”【31†L286-L295】.  Some projects address this: the Harris Matrix Composer (HMC) planned a powerful ArcGIS interface to “select units of the Harris Matrix” and visualize them on plans【46†L1-L4】.  Recent development (HMC+) adds a shapefile importer that can validate and import hierarchical stratigraphic data (groups/periods)【59†L1419-L1427】.  Meanwhile, field-logging apps like QField offer **instant matrix generation** during recording: QField supports a built-in “Harris matrix” view linked to stratigraphic entries and georeferenced images【67†L1-L9】.  (By contrast, some tools like iDig auto-generate the matrix in-app, while others force export to offline tools【67†L7-L15】.)  

For **chronological modeling**, the trend is toward standardized data exchange.  The research community is developing a Harris Matrix data package (HMDP) specification (Costa et al. 2023) precisely to facilitate exports to Bayesian dating tools.  Our sources note that obtaining stratigraphic archives “in suitable form for input into… OxCal” is a current project priority【35†L308-L317】.  Thus, next-gen software should ideally export both the relative sequence and phase definitions (with confidence bounds, sample associations, etc.) directly into OxCal/BCal formats, avoiding manual table wrangling.  

**3D data and photogrammetry** are also entering the workflow.  Archaeologists routinely capture detailed 3D site models (laser scans, UAV photogrammetry), but these are typically separate from the matrix.  Emerging systems aim to link them: for example, advanced 4D stratigraphy platforms visualize digitized strata volumes in a Unity 3D scene with interactive time-slider tools【60†L85-L93】【62†L1-L4】.  Such efforts show the value of integrating **matrix + 3D model + database**: the literature observes that “an integrated information system including data storage, 3D visualization and… stratigraphic analysis… is… under study and development”【60†L85-L93】.  Practically, archaeologists would like a matrix tool that can, say, highlight the 3D extent of each context when clicked, or allow phasing of 3D layers in synchrony with the matrix.  As one review notes, most GIS and CAD packages still “act as efficient overlays” but lack temporal analysis, so HM tools should bridge that gap【31†L318-L327】【60†L85-L93】.  

In summary, the **industry is moving toward interoperability** via common formats and shared data.  Legacy HM outputs (flat images or prose) are being replaced with exchangeable tables/graphs (e.g. CSV, JSON, or Frictionless HMDP packages) and web APIs.  Open-source bridges (e.g. the Python `hmdp` tool) are appearing to link excavator databases, QGIS, and OxCal.  These efforts indicate that archaeologists want stratigraphic software to plug into GIS and chronology software, aligning with FAIR data principles【35†L308-L317】【67†L7-L15】.

## Latent Needs and Innovative Opportunities  
Beyond expressed requests, modern tech enables new features archaeologists may not realize are feasible.  For instance, most HM tools still use tree or frame-based data models, but **graph databases** (Neo4j, RDF triples) could more naturally store contexts and relationships.  A graph backend would allow semantic queries across sites: e.g. “find all contexts containing medieval ceramics in Phase 3”, linking to Linked Open Data and museum catalogs.  Coupled with SPARQL or Cypher, this could let analysts explore stratigraphy in rich ways (a capability hinted by calls for “data storage and… spatial analysis” integration【60†L85-L93】).  

**AI and machine learning** could transform stratigraphic analysis.  For example, computer vision or clustering algorithms can **automatically suggest stratigraphic units** from site imagery.  Recent research applied K-means color clustering to trench photos to delineate layers, offering a “prompt guide” for archaeologists【64†L69-L77】.  In practice, an HM tool could let a user upload orthophotos or 3D point clouds and have AI identify probable deposit boundaries or anomalies.  Similarly, NLP could assist by ingesting context descriptions and flagging potential relationships (e.g. “pit cuts” vs “fills”) to prefill the matrix.  Such AI augmentation would reduce manual drudgery: the cited ML study notes archaeology has “never used” these techniques yet, but demonstrates their potential【64†L69-L77】.  

On the **visualization front**, WebGL and game engines open up immersive interfaces.  A next-gen stratigraphic tool might run in the browser with 3D WebGL: imagine a navigable 3D excavation model where the Harris Matrix floats beside it, and selecting a box in the diagram highlights its volume in space.  The recent “Castulo 4D” project implemented exactly this using Unity【62†L1-L4】.  Web-based 3D (using libraries like three.js or Babylon.js) could make HM diagrams explorable – layers could expand/collapse in 3D, temporal sliders could animate site formation, and VR/AR interfaces could project the stratigraphy in situ.  

Other latent innovations include **real-time multi-user collaboration** and versioning (version control for the HM diagram), social/annotation features (comment threads on relationships), and rich multimedia integration (attaching photos, soil analyses, or VR panoramas directly to matrix nodes).  Modern web frameworks (React, WebSockets, etc.) and cloud databases make these achievable.  Finally, features like **integrated data validation and reasoning** could be built: automated consistency checks (cycle detection, logical constraints from Harris’s laws) already exist, but could be extended with rule engines or machine reasoning to detect more subtle errors or suggest phase-groupings.  

In summary, archaeologists’ *latent* needs include an **unified, graph-driven data model** bridging temporal, spatial and artefactual information; AI-powered auto-indexing of layers; and immersive 3D or web-based interfaces that integrate the stratigraphic graph with the spatial site model.  These go beyond traditional HM tools but are within reach of today’s technology (graphs, ML, WebGL)【60†L85-L93】【62†L1-L4】【64†L69-L77】.
